Web review: reminders

There are a few things that bear reminding:

  • file and folder names cannot contain:
  • capital letters
  • spaces
  • special characters:
! @ # $ % ^ & * ( ) + { } [ ] : ; " < > ? |
  • acceptable non-alphanumeric characters: . _ - ~
  • default page: index.htm, index.html or index.shtml (or whatever your server default is)

As this is a web development course, let’s talk a bit about coding patterns & behaviour.

Comments

Comments in your code are great! For you, anyone you work with or anyone looking at your site, both today and especially in the future.

Browser Tools

Lots of options

These days there are lots of browser-based tools available for web developers. Initially, Mozilla (Firefox) built in some basic tools and their plug-ins centered around web development.

Now, Webkit (Chrome, Safari) and Mozilla (Firefox) have excellent tools built-in. Some need to be “turned on” but others are built in by default.

The tools are enhanced by their organizations and are ever-increasing in capability and what they can do.

Do not underestimate how powerful these tools can be! I can guarantee these tools will get you out of trouble, or solve a problem you didn’t even know you had.

What kinds of things do these tools help you with?

  • Checking validity of your code
  • See errors in rendering
  • Use the console for Javascript output (remember to code your error throwing!)
  • Slow loading resources
  • Large resources
  • External bottlenecks
  • Rendering issues
  • … much more!

While we will look at Chrome and Safari, the focus will be Firefox in COMP1950 so that we are all working from the same platform.

Remember: to check in all browsers when taking a look at your site for production.

Standards

What are standards? Why are they important?

What are some examples of standards in real life?

  • Lightbulbs
  • Grades of gasoline
  • Train rail widths
  • TV Broadcast signals

Why use them?

  • Interoperability
  • Predictable results/dimensions/values
  • A new specification is not needed for each new manufacturing run
  • Need a new microwave? You can count on:
  • Plug/Voltage/Power
  • Unit size
  • Operation

Aging badly

When you don’t support standards, you or your clients suffer for it later on. I’m looking at YOU, IE6!

Web Standards

When you build without standards, you are investing in the here and now, without thinking of tomorrow.

Standards development works both ways:

  • Browser manufacturers need to code to a predictable standard
  • Developers code to the standard they expect within the language (HTML, CSS and JS) for a predictable result

Problems occur when either the browser manufacturers or the develoers stray from the standard.

Who sets the standard?

The World Wide Web Consortium (W3C) is responsible for maintaining the Web Standards, which has a gigantic effect on the world.

W3C Goals

  • Universal Access: To make the Web accessible to all by promoting technologies that take into account the vast differences in culture, languages, education, ability, material resources, access devices, and physical limitations of users on all continents;
  • Semantic Web: To develop a software environment that permits each user to make the best use of the resources available on the Web;
  • Web of Trust: To guide the Web’s development with careful consideration for the novel legal, commercial, and social issues raised by this technology.

Do standards actually matter?

  • Your pages will be viewable in the largest number of browsers
  • The largest number of devices will be able to render your pages
  • Will gracefully degrade as support for CSS or other presentation layers are absent
  • Most importantly: Standards Mode

Semantics

Okay, so Standards sound like a good idea, but one of the W3C’s goals is something called Semantic Web. What is that about?

Semantic Markup

Semantic markup is simply…

The practice of using HTML tags that convey meaning of the content within

What does that mean in practice? Most of it is pretty basic. Things like:

  • H1 … 6 tags for headings
  • Paragraph tags instead of double BR tags
  • CSS instead of FONT
  • Tags that describe content (like code, abbr or cite)

Logical vs. Physical Styles

While the content looks the same, there are differences between logical and physical styles.

Easy to remember:

Physical styles define what content looks like. Logical styles define what content is.

Important for screen readers and other access devices.

Some physical styles:
  • b - Bold
  • i - Italic
  • s - Strikethrough
  • u - Underline
Examples of logical styles:
  • code - Code sample
  • kbd - Keyboard entry
  • samp - Programming sample
  • var - Variable
  • cite - Citation
  • dfn - Definition
  • abbr - Abbreviation
  • acronym - Acronym
  • del - Deleted text
  • ins - Inserted text

Key Advantages

  • Extracting important pieces of information is simple (CITE or CODE tags for example)
  • Search engine optimization
  • Big win for jQuery
  • Screen readers

Standards: The need for XHTML

So, how did we get to XHTML?

  • HTML was a language created at the dawn of the Web, using SGML (Standard Generalized Markup Language) as a template
  • Through a flurry of iterations, HTML 1.0 moved to 4.01 fairly quickly over 5 years or so
  • Despite it’s reputation as a loose language, HTML 4.01 can be written strict just like XHTML and HTML5 can
  • The problem was not with the language, nor with the authors. It was with the browsers.
  • Browser manufacturers created User Agents that would read code, interpret it, fix any mistakes and render a page, without any visible errors.
  • It was this that led to coders exploiting problems in browsers and creating pages that broke when new specs or updated UAs came out
  • Millions of pages are being produced with sloppy code because the UAs let it happen
  • The web is becoming filled with Tag Soup that makes it hard to index, with search tools having to trust things like META keywords and descriptions
  • A better solution is required: Get Strict!

Clean up the mess

  • After years of HTML development, 4.01 became the “Last Official Release”
  • The Web Working Group had taken a look at XML (Extensible Markup Language) and liked what they saw: a requirement for valid markup in order for an XML application to work
  • Why not take the ruleset from XML (lower case attributes, everything in quotes, close every tag) and apply it to HTML? Yay!
  • That should fix everything, right? Uh, no.
  • Original authors of XHTML wanted to force strict coding rules so that the Web could be seen as a giant application.
  • Reliable code on every page meant you could use the data anywhere
  • However, draconian error handling would cripple the web

On toward XHTML2

  • It was that last issue and the next one that were the main tipping points for the demise of XHTML, although many of the people in the Web Working Group didn’t know that yet
  • Between 2002 and 2006, a significant amount of development was attempted for XHTML2
  • An early decision was to abandon backward compatibility with XHTML1 and HTML4.x leading to significant controversy
  • What does that mean? “abandon backward compatibility”?
  • Deprecated tags would no longer work
  • All “old” code would have to be updated if you wanted to deliver XHTML
  • Browsers would have even more work to do when rendering pages

Discussion and disruption

  • Some of the more contentious issues were split out into modules to be worked on by the W3C XHTML Working Group
  • At one point, a question was asked:

Shouldn't we be taking into account the billions of pages that are already out there, make the upgrading part easy and focus on the kinds of things people are actually doing today? Maybe not exactly that, but close

  • A vote was called and the motion was defeated. But the people who thought it was a good idea wanted to move forward
  • As a result - they created the WHATWG (Web Hypertext Application Technology Working Group) and began working on a competing specification, which was originally called Web Applications 1.0

XHTML - A review

X = eXtensible

  • HTML reformulated as an XHTML application.
  • First major update to HTML since the 4.0 specification was released in 2000.

Is there any reason why we should know, or code in, XHTML?

  • HUGE codebase of XHTML pages out there
  • Understand XHTML in order to convert it
  • Better, more predictable coding patterns
  • “verifiable” or “well-formed” code can be produced
  • Forces a higher standard of HTML
  • Porting to HTML5 is easy (if you also observe semantic guidelines)

XHTML Rules:

All elements must be properly nested

This is not valid: 

<p><b>Bolded Text</p></b>

This can be a problem with things like Lists ol and ul. Especially when closing the List Item li tag that contains the parent tag.

All tags must close

Example: 

<p>Some text here</p>

Empty Elements are closed as well: 

<br /> or <img src="#" />

An extra space is required before the slash to work with all browsers.

All tag names must be in lower case

This is not valid: 

<Img Src="some_image.gif">

Instead, must be:

<img src="some_image.gif">
Attribute names are always in lower case 
<p ALIGN="center"> should be <p align="center">
All attributes must be quoted

Invalid:

<body bgcolor=black>

Valid:

<body bgcolor="black">
Attributes cannot be minimized

Will not work in XHTML:

<input type="checkbox" checked>

The valid code is: 

<input type="checkbox" checked="checked">
Name Attribute is no longer used. Replace with id.

This will not validate:

<img src="picture_off.gif" name="picture_off" />

Instead, you will need to use: 

<img src="picture_off.gif" id="picture_off" />

In order to work with new and older browsers, you can use both name and id (however, you will get a warning).

Mandatory Elements

Every XHTML document must have these elements:

<html>
<head>
<title>
<body>

The doctype declaration must be there, but it is part of the document itself rather than an element of the document.

Optional XML declaration

Not required, but good practice:

<?xml version="1.0" encoding="ISO-8859-1"?>
DTD: Document Type Definition

A DTD is a description of the document, and the legal definitions that fall within it.

The DTD defines the syntax of the page in SGML. This translates to: Because XHTML is just the application of XML to HTML, you need to tell the browser (the interpreter) what rules to use when interpreting the code on the page.

Three kinds of DTD: Strict 
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Strict//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-strict.dtd">

Mostly used for technical documents or content that requires very little markup. This is the original intention of the W3C”s recommendation.

Transitional (most common) 
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

This is by far the most common DTD in XHTML. It supports most HTML features and was created to provide a middle ground for users trying to convert from HTML to XHTML in stages.

Frameset 
<!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Frameset//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-frameset.dtd">

Same as Transitional, but allows the use of frames

Example of an XHTML Document <?xml version="1.0" encoding="ISO-8859-1"?> <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> <html> <head> <title>COMP 1950 - Intermediate Web Development and Design</title> </head> <body> <h1>Welcome!</h1> <p>This class will begin at 6:30 each Tuesday for 12 weeks</p> </body> </html>

HTML5

HTML was born out of frustration with languishing XHTML2 development, which went on for years after XHTML 1.0 debuted.

  • The last official spec of XHTML had come in 2000 and by 2005, there was little forward movement, with the development bogged down in committees and problems reaching consensus
  • There were two distinct camps: Strict and Non Strict (overly simplified terms)
  • After a series of decisions against a more forgiving, backward compatible version, a number of HTML Working Group members created their own group: WHATWG
  • After 3 years, Web Applications 1.0 gained considerable momentum when Mozilla, Apple and Opera joined the Working Group and requested that it become part of the W3C’s HTML Working Group
  • In 2007, the W3C HTML working group adopted the development, renamed it HTML5 (a successor to HTML 4.01) and released a Public Working Draft in 2008
  • As of today, HTML5 is a Living Standard, and is now just called HTML
The basics of HTML5

Some of the most important parts of HTML5 are:

  • Backwards compatible - there will be an attempt to make the system compatible
  • XML can be included as a markup language from within HTML5
  • Ultra simple Doctype: 
<!DOCTYPE html>

One of the guiding principles: Semantic Coding - giving meaning to your content with your code.

While these may seem insignificant, but they are great strides forward.

Some of the most common new elements:

This a tag representing a document or application section:

<section>

This tag is an independent piece of content in a document:

<article>

For content “only slightly” related to the content:

<aside>

Used for the header or footer on a page:

<header> or <footer>

Represents an area for navigation:

<nav>

For when you want to include an element that is related to the content in an image, sound, video or other: 

<figure>

There is a list of additional tags that will continue to refine what will be possible with HTML5: audio, video, embed, mark, meter, progress, time, canvas, command, datagrid, details, datalist, keygen, bb, output, ruby

Input & Type elements will also have new attributes that allow for greater Form functionality:

datetime, date, month, week, time, number, range, email, url, search, color

There are a huge list of additional attributes that have changed, allowing for considerable agility with web development

There have also been a few elements removed, such as: big, font, strike, and a few more

There are also a large number of attributes that have been removed, such as: align, background, border, height, size, type, width and more
